Introduction

Welcome back! Until now, you’ve focused mainly on general programming concepts and how they work in JavaScript, as well as how to interact with the DOM. But there’s still one last building block to discuss—asynchronous behavior.

Did you know that pretty much every modern app uses some form of asynchronous behavior? Take any social media application, for example. Have you ever noticed how your feed appears to constantly refresh itself before you even get to the newer posts? This is because the app asynchronously loads data in the background while you scroll. This can hugely improve the user experience of your app, making it smoother and more intuitive.

In this Exercise, you’ll be learning all there is to learn about asynchronous behavior (and how it differs from synchronous behavior), as well as how you can load data from external sources like APIs asynchronously in your own app. Ready to get started?

APIs

What is an API? Well, say you wanted to import tweets from Twitter and display them somewhere on a website you’ve built. To do so, you’d need to ask Twitter to give them to you. Once Twitter verifies that you’re the owner of the tweets, they’d hand them over. This whole process is taken care of by an application programming interface (API). Think of it like a go-between, a kind of helper between your app and the app or database you want to connect it to.

The set of techniques you’d use to have this back-and-forth interaction is holistically referred to as “Ajax”: more on this later.

So, an API is a middle-man, a defined way for two or more computer programs to interact with each other, like an app and a server, for instance.

In the next Achievement, you’ll learn how to build your own API. There are, however, a great many APIs out there already. Oftentimes, web apps have APIs that allow developers to build their own extensions. For example, Google Search has an API that lets you search from your own app.

If you’re interested in learning more, check out this list of helpful and fun APIs for developers.

When it comes to loading data from an API using JavaScript, you’ll commonly use what’s called asynchronous code.

Synchronous vs. Asynchronous Code

The easiest way to understand the complexities of asynchronous code is to look first at synchronous code: what exactly it is and how the two differ.

Synchronous Code

Synchronous code entails everything you’ve worked with so far in this course and involves executing your code in order. Take a look at the following JavaScript code:

// "load new tweets" button has been clicked
doSomething();
displayLoadingGif();
fetchNewTweets();
displayNewTweets();
stopLoadingGif();
doSomethingElse();

Each of these functions will be called one after another in the order they appear in the file. No matter how much time a function needs to complete, JavaScript will wait until it’s finished before moving onto the next line. This is what makes the code synchronous. It’s also the most straightforward way to code.

However, what if you didn’t know for sure when a certain piece of code was going to complete its task—for example, fetching a list of tweets (fetchNewTweets())? You only know that the task will be completed some time in the future. If fetchNewTweets() were to be executed synchronously, any code that came after it (e.g. doSomethingElse()) would be paused until fetchNewTweets() was finished and the tweets were fetched.

For code that’s executed instantaneously, this wouldn’t be a huge problem, but what if it took 2 or even 30 seconds for the tweets to be fetched? The rest of the application would be forced to wait, unable to do anything until the tweets are fetched, effectively rendering your app useless during that time, similar to the left-hand example in the gif below:

Figure 1

In the image above, the loading time has been exaggerated, but you can still see this type of behavior isn’t conducive to most modern apps. Allowing your app to stall while it’s waiting for an action to be completed simply isn’t good practice.

Imagine being the person using the left-hand app: you click to refresh your tweets, but because of a spotty internet connection, the app takes a while to refresh, during which time, you can’t click on any “like” or “share” buttons, almost as if the code for these buttons didn’t exist. It does this because everything is set up synchronously—no new code can be executed within the app until the current function has finished executing. As the app is still in the middle of running fetchNewTweets(), it can’t simultaneously try to run, for instance, likeTweet() or shareTweet().

By likening this to a real-world example, you can see how even more ludicrous this type of behavior is. Say that you ordered a pizza. You don’t know for sure when it will arrive—simply that it will be “some time in the future” (just like the tweets will be fetched “some time in the future”). In a synchronous world, you’d have to wait on your doorstep doing nothing until the pizza arrived. Not exactly efficient! In an asynchronous world, however, you’d be able to do other actions while you wait, such as “watch TV,” “get drinks,” or “talk to friends.”

This “asynchronous” code is exactly what you’re going to be learning about and practicing in this Exercise. Employing it isn’t just beneficial to the apps you’ll be creating—it’s downright essential.

Asynchronous Code

Asynchronous code is code that can be executed at any point in time and that doesn’t require the lines of code that follow it to wait. Think back—you’ve actually written code like this before! Remember your event handlers?

doSomething();
element.addEventListener('click', displayHelloMessage);
doSomethingElse();

The event listener above “listens” for the user to click, then does something in response. But imagine if this worked synchronously. The browser would reach this line in the code and be unable to proceed further until the event listener code was completed (in other words, when a user clicked). This isn’t what you want at all; rather, you want the rest of the code to keep running normally whether or not a user clicks. Your event listener needs to run at some point in the code, but not necessarily where it’s been written.

Functions like displayHelloMessage in the above example are called callbacks. This term is used to describe functions that are called when something asynchronous happens (in the case above, once the user clicks on the element). Consider, for instance, a timer:

function runThisLater(){
  console.log('Burgers');
}

console.log('Pizza');
setTimeout(runThisLater, 1000);
console.log('Salad');

In the example above, runThisLater is the provided callback function, which will be executed in exactly 1000ms. Again, you wouldn’t want to require all the code that follows this line to also wait 1000ms before being executed. That’s why you, instead, call the runThisLater function asynchronously and independently of the code that follows. If you were to execute the code above, “Pizza” and “Salad” would be logged instantly. Then, after one second, “Burgers” would be logged. Rather than wait idly for one second, the browser would continue executing the code following setTimeout(runThisLater, 1000);.

To return to the tweets example from earlier, an asynchronous version of the code could like something similar to this:

// "load new tweets" button has been clicked
doSomething();
displayLoadingGif();
fetchNewTweets(function(){
  displayNewTweets();
  stopLoadingGif();
});
doSomethingElse();

The idea here is that when executing fetchNewTweets() in an asynchronous fashion, the browser won’t wait for fetchNewTweets() to finish executing before executing the line doSomethingElse(): the browser will move onto it immediately.

Asynchronous code can be extremely effective in the right use cases; however, just like synchronous code, it comes with its own set of problems. For instance, see if you can figure out what order these console logs will appear in:

let count = 1;
function increaseCount() {
  count = count + 1;
}
setTimeout(function () {

  console.log('first call', count);
  increaseCount();
}, 200);
setTimeout(function () {

  console.log('second call', count);
  increaseCount();
}, 100);

  increaseCount();
}, 500);

This was a rather simple example, but you can see how things could become quite complicated when more variables and functions are involved. Imagine if you were to program something to happen immediately after a user clicks a button. Now, imagine that the user clicks the button multiple times very quickly. What would happen? Your code needs to be able to handle situations like this and behave accordingly.

Given what you’ve already learned about APIs, do you see why asynchronous code would be the ideal kind of code to use with an API? If you’re using an API to send messages back and forth with a server, or some other program, you would want all that to happen in the background while your code keeps running.

Remember—the techniques that manage the back and forth interaction with the API are called “Ajax.” In the next section, you’ll be looking into three topics central to Ajax:

  • Promises: a more-structured and organized way to code asynchronously than callbacks
  • JSON: the most common data format used when exchanging data with external sources
  • HTTP: a protocol for making requests

Promises

Promises offer a cleaner way of writing asynchronous code. The promise is what holds that code that needs to be executed asynchronously. For example, a promise could fetch data, listen for an event to occur, and more. Take a look at the following code snippet, which shows how promises can be used in code:

console.log('some piece of code');
examplePromise.then(function(result){
  console.log(result);
}).catch(function (error) {
  console.error(error);
});
console.log('another piece of code');

Let’s break this code down piece by piece:

  • In the above example, examplePromise is the promise. This is the thing that may or may not be fulfilled; the fetching of data, an event occuring, etc.
  • If the code in the promise is successfully completed (i.e., if the promise is “resolved”), the code within .then() is executed.
  • When a promise is resolved, it can optionally return a value. This piece of data is assigned to the parameters of the callback function in .then(). In the code above, result is the only piece of data returned, and it’s assigned to the parameter in the callback function.
  • If the promise doesn’t go as expected (i.e., if the promise is “rejected”), the code within .catch() will be executed.
  • When a promise is rejected, it can optionally return a value that will be assigned to the parameters of the callback function in .catch(). In this case, there’s only one parameter: error.

The above example only demonstrates what would happen after a promise is either resolved or rejected. But what would the promise itself look like? Let’s take a look at an example:

let examplePromise = new Promise(function (resolve, reject) {
  let sum;
  setTimeout(function(){
    sum = 5 + 6;
    if(sum > 10) {
      resolve(sum);
    }else{
      reject('The promise has been rejected');
    }     
  }, 2000);
});

While the above example might not be practical, it does show the basic structure of a promise. A promise is formed by creating an object out of the Promise class. This will be similar to what you already learned about creating objects in Exercise 1.3: JavaScript Basics Part 2, but it differs in that it’s being created from a class. Think of a class as a template. The Promise class is a template providing everything a promise would need to function properly in JavaScript.

To get a copy of that template, use the new keyword followed by the class name, for instance, new Promise. Within the parentheses are any initial properties that make that copy unique compared to other copies you can create from a specific template. Here, for instance, the Promise class will take only one property, which is a function. That function will have two additional functions inside of it—resolve and reject. Here’s what they mean:

  • resolve: This function is called to inform the browser that the promise has been resolved. You can optionally pass a value as illustrated above, where it will then be present in the callback function within .then(). Say, for instance, that you’ve created a promise that fetches a feed from a news website. If the fetch were successful, you would pass everything within the feed as an array, string, etc. into the resolve function call.
  • reject: This function is called to inform the browser that the task within the promise didn’t go as planned (i.e., it was rejected). Think of it as a way to handle errors. To continue with the news feed example, you could have an if statement that checks whether the news feed array is empty. If it is, you could pass the string “News feed unavailable” into the reject function call, where it could then be used within the .catch() statement.

Finally, the setTimeout() function is used to simulate a task that would take two seconds to finish. Normally, you wouldn’t know exactly when the task would finish; rather, you’d know that it would be finished “some time in the future.” But for the purpose of this example, the setTimeout() function can simply represent the delayed completion of a task, which is the main point of asynchronous behavior.

Combining the two previous code snippets, you then get this:

let examplePromise = new Promise(function (resolve, reject) {
  let sum;
  setTimeout(function(){
    sum = 5 + 6;
    if(sum > 10) {
      resolve(sum);
    }else{
      reject('The promise has been rejected');
    }     
  }, 2000);
});

console.log('some piece of code');
examplePromise.then(function(result){
  console.log(result);
}).catch(function(error){
  console.log(error);
});
console.log('another piece of code');

This looks great. But there’s still one small issue—the code inside the promise creation block will technically be executed before console.log('some piece of code'). While there’s nothing syntactically wrong with this, it’s better practice to only execute this code the moment the promise is settled (resolved or rejected).

To ensure this, you can wrap the promise creation code block within a function that returns the promise. Then, when you want to listen to the promise, you can simply chain .then() and .catch() into the function call. Let’s see what this would look like:

function examplePromise(){
  let promiseToReturn = new Promise(function (resolve, reject) {
    let sum;
    setTimeout(function(){
      sum = 5 + 6;
      if(sum > 10) {
        resolve(sum);
      }else{
        reject('The promise has been rejected');
      }     
    }, 2000);
  });
  return promiseToReturn;
}

console.log('some piece of code');
examplePromise().then(function(result){
  console.log(result);
}).catch(function(error){
  console.log(error);
});
console.log('another piece of code');

Because of the .catch() block, even if the promise is rejected (e.g., if the value of sum is less than or equal to 10), something will still happen. In this case, the message “The promise has been rejected” will be displayed, which is then propagated to the error parameter in the callback of the .catch() block that’s chained to the promise. This ensures that the correct actions will be taken whether the promise is resolved or rejected.

Before finishing up this discussion on promises, let’s quickly highlight two important ways promises give you more flexibility than callbacks:

  1. You can pass as many values as you want in your resolve and reject function calls, and all of these values will be available in the callback functions of your .then() and .catch() statements.
  2. You can call resolve or reject multiple times when creating promises; for instance, if you have multiple success/failure paths. Take a look at the example below:
let examplePromise = new Promise(function (resolve, reject) {
  .. some code ..
  if(.. Condition 1 ..) {
    resolve('The promise has been resolved');
  }else if(.. Condition 2 ..){
    anotherPromise.then(function(){
      .. some code..
      resolve(value1, value2, value3, array1);
    }).catch(function(){
      reject(error1);
    });
    reject('The promise has been rejected');
  }else{
    reject(error2, specialValue);
  }
});

Not only are promises more flexible than callbacks—they’re also a lot more powerful. Later in the Exercise, you’ll see how they’re useful in real-world scenarios (since all of the above examples have been hypothetical).

JSON

One of the most important aspects of Ajax is JSON, which stands for “JavaScript Object Notation.” JSON is a special light-weight language for transferring data back and forth between a server and a web app. It’s not the same exact thing as JavaScript, but it’s written in syntax that looks identical to JavaScript:

{
  "name": "John",
  "age": 31
}

JSON was born out of JavaScript, which is why the two are so similar. Because of this, any object you define in JSON will also be a valid JavaScript object.

But the reverse isn’t always true. There are some features in JavaScript that are not allowed in JSON, mainly:

  • Functions
  • The undefined value (only null is allowed)
  • Single quotes (only double quotes are allowed)

Also, in JSON, all object keys must be written with double quotes " ".

Considering that there are relatively few restrictions, working with JSON in JavaScript is quite easy. It still supports strings, numbers, booleans, null, arrays, and sub-objects, and you can assign as many objects inside objects as you want (the term for this is arbitrary depth).

TIP!
Not only was JSON’s design based on JavaScript, it’s also well-supported in just about every other programming language, making it a great format for interactions between JavaScript and servers.

So, why should you care if something is JSON or just a “regular” JavaScript object? Why do you need JSON in the first place?

Well, if you’re going to be sending and receiving data from an external source, you’re almost guaranteed to need JSON. Just about any server you work with in the future is going to use JSON. This is because servers can only store data in plain text, making the more advanced functionality of other programming languages incompatible. Take a look at the example below (you can use Chrome’s developer console to run the code and check the output):

let jsonStringFromServer = '{"name":"Lisa","age":27}';
let lisa = JSON.parse(jsonStringFromServer);
console.log(lisa.name, lisa.age); // 'Lisa', 27

Notice how the value of jsonStringFromServer is actually a string rather than a JavaScript object? This is, once again, because all JSON data has to be in plain text.

Fortunately, JavaScript comes with a built-in function for converting this stringified object into a real object: the JSON.parse() function. “Stringified,” here, means that the code simply looks like an object but is actually a string. In the example above, you wouldn’t be able to use dot notation to call values from jsonStringFromServer as it’s not an object. However, after converting the string with JSON.parse() (and storing it in the variable lisa), you’ll be able to use dot notation just fine.

The same holds true when you need to work in the opposite direction—from an object to a JSON string (use Chrome’s developer console to test the code):

let john = {
  name: 'John',
  age: 31
};
let jsonString = JSON.stringify(john);
console.log(jsonString); // '{"name":"John","age":31}'

This is useful when you want to send JSON back to the server (e.g., to create or update a record). Note, here, that all non-supported properties will simply be ignored when calling JSON.stringify():

let complexObject = {
  name: 'John',
  getAge: function () {
    return 20;
  },
  note: undefined
};
console.log(JSON.stringify(complexObject)); // '{"name":"John"}'

You can also stringify arrays, which basically works the same as it does for objects:

let list = [1, 2, 3];
console.log(JSON.stringify(list)); // '[1,2,3]'
let otherList = JSON.parse('[3,4,5]'); // [3, 4, 5]

Note how the stringified array doesn’t have any spaces. Stringifying an array (or an object) will turn it into a string the way your computer sees it. All spaces, tabs, or line breaks you add to arrays while writing your code will be ignored in the stringifying process, as they only matter for code readability but have nothing to do with the array itself.

TIP!
You might remember from an earlier Exercise that copying an object or array isn’t the most straightforward of processes. JSON can actually help with this! So long as the object/array in question doesn’t contain any functions, you can use the code let copy = JSON.parse(JSON.stringify(obj)) to create a complete copy of the given object.

HTTP Continued

You first learned about HTTP back in your Intro to Frontend Development course, but there’s actually quite a bit more to it that you didn’t cover at the time, especially when it comes to working with Ajax. Let’s take a look at what else you should know about this important protocol before you can use it with Ajax.

HTTP Methods

HTTP supports different methods for making requests to the server. Although these methods are sometimes presented as nouns, they can also be referred to as HTTP verbs. The most important ones you’ll want to know are: GET, POST, PUT, and DELETE.

You’re already quite familiar with the GET command, though you might not realize it. Every time you open up a web page, the browser is using the GET command to load the page. This is the default request, and it does pretty much what its name suggests—“getting” (or “reading,” or “fetching”) some kind of content.

With JavaScript and other tools, you can make other types of requests to web servers:

  • POST is used to do something, such as creating a record or sending an email.
  • PUT is used to update something, such as a record that was previously created via POST (this is similar but different to another command, PATCH).
  • DELETE is used to delete something.

These four methods form the acronym CRUD: Create, Read, Update, and Delete. When someone talks about “implementing CRUD” for something, it means enabling the user to create/read/update/delete a given resource, such as an item on a to-do list. In terms of HTTP, then, you can post (create), get (read), put (update), and delete (which is, well, delete).

Difference Between PUT and PATCH

The main difference between PUT and PATCH requests is how the server processes the request to modify the relevant resource:

  • In a PUT request, the new resource is considered to be a modified version of the original resource and overwrites it.
  • With PATCH, the new resource updates the old one by adding, removing or modifying the existing data to produce a new version.

Another difference is that when you want to update a resource with a PUT request, you have to send the full item payload as the request, whereas with PATCH, you only send the parameters and fields that will update the old version.

Let’s take a simple example to understand the difference between the two. Suppose you have stored the first names and phone numbers of your friends in a web server. One of your friends, Ron, has recently changed their number, so you need to update that record.

If you want to use a PUT request, you should send something like this:

{ "name": "Ron", "Phone": "+4915123456789" }

Although you only want to update the phone number of Ron, you need to send both parameters (name and updated phone number).

What if you choose to use PATCH? Now, you just have to send the new phone number as the parameter, without sending the name. With PATCH you send something like this:

{ "Phone": "+4915123456789" }

To sum up, when you need to replace an existing resource entirely, you should use PUT. When you want to do a partial update, you should use PATCH.

Treat these recommendations more like guidelines than strict rules. When someone creates an API for developers to work with, they’re encouraged to follow these principles, but there’s nothing technically stopping them from choosing different behaviors for these methods.

When talking about HTTP requests, developers will often talk about “posting to the server” or “patching a resource.” These types of phrases are shorthand for “making a request to the server with the POST or PATCH method.”

HTTP Status Codes

Every HTTP request will result in a numeric status code. These are meant to indicate the status, or “result,” of a request (e.g., whether or not an error occurred, etc.), and they come in a wide variety to account for the plethora of possible errors one might encounter. In fact, you’re probably already quite familiar with one—the infamous “404 Not Found.”

Some of the other important codes you’re likely to run into include 500 “Internal Server Error” (generic error), 200 “OK” (non-specific success), 301 “Moved Permanently” (redirects the user to another URL) or 401 “Unauthorized” (problem with the client’s credentials). For a comprehensive list of status codes, you can check out Wikipedia.

HTTP Media Types

Every HTTP response has its own media type (also called its MIME type) that indicates what type of data the response contains. Common media types include:

  • text/plain: plain text with no specific content structure or markup
  • text/html: HTML content
  • application/json: JSON content
  • application/xml: XML content

For example, a request to “index.html” would result in a response with the media type text/html, but a request to “https://www.google.com” would also return text/html in the form of an HTML file to your browser, even though it doesn’t have a “.html” extension.

What is Ajax?

The term Ajax is an acronym for “Asynchronous JavaScript + XML.” Ajax doesn’t describe a specific technology, rather, a programming concept.

To code using Ajax principles is to create a site or app that can load data asynchronously. Rather than requiring a page refresh to display new information, the page can, instead, be altered dynamically (think automatically updated feeds, menu bars and drop-down tabs, pop-up modals, etc.). Much of this is, in essence, what modern JavaScript applications already do, which is why the term Ajax tends to only be used for the “loading data asynchronously” part of the definition.

TIP!
You might be wondering what the “XML” part of the Ajax description means. XML is a data-exchange format. Though it’s mostly been replaced by JSON, you’ll still find it floating around in certain places. If you’re interested in reading more about XML, check out the Resources section at the end of this Exercise.

Ajax with Fetch

fetch is a simple function introduced in ES6, 2015’s version of JavaScript. It’s a built-in function that allows you to get, or “fetch,” data asynchronously from external data sources such as Twitter, Youtube, Facebook, and the like. To do so, it uses the JavaScript promises you learned about earlier.

This function will prove essential for your Achievement project, where you’ll need to fetch a list of Pokémon from an external data source (the Pokémon API). Take a look at the example below:

fetch('https://pokeapi.co/api/v2/pokemon/').then(function (response) {
  return response.json(); // This returns a promise!
}).then(function (pokemonList) {
  console.log(pokemonList); // The actual JSON response
}).catch(function () {
  // Error
});

More Practicing
We’ve created this Fetch and Promise repl to show you a simple example of how fetch and promises work together.

The fetch() function in your Pokémon example is designed to return a promise, which means you can chain .then() and .catch() statements in it. When the promise returned by fetch() is resolved, the response from the external source will be passed to the callback function in the .then() block.

But wait! What about the second .then() block in the code above? This is actually a nifty trick you can employ when working with promises, called “Promise Chaining.” Not only does the original fetch() function contain a promise, but so does the callback function inside of its .then() block. Thus, the second .then() statement will contain the callback function for this second promise.

Let’s walk step by step through the Pokémon example:

  1. First, fetch('https://pokeapi.co/api/v2/pokemon/') sends a message to the specified API, requesting the list of Pokémon.
  2. When the promise is resolved, an HTTP response containing the list of Pokémon is passed to the response parameter in the callback function of the first .then() block.
  3. If you tried to log response into the console now, you’d get the object [object Response].
  4. If you were to expand this logged object, you’d see its properties/keys, among which you’d find the json key, which holds a function that parses the response body into JSON data. This is how the response object returned by fetch() is designed—rather than directly getting the JSON data, it needs to first be parsed via the provided json() function.
  5. You now know that the json() function returns a promise. This means that return response.json(); will return a promise object, which is why there’s a second .then() block chained onto the first .then() block. If the promise within response.json() is resolved, any data passed in the resolve function will be available in the next .then() block. In the case of the above example, this means that pokemonList will contain an array of JSON objects, each representing a single Pokémon.
  6. Finally, the .catch() block handles any errors that may occur if the promise is rejected.

And that’s it! Keep in mind that fetch('https://pokeapi.co/api/v2/pokemon/') is shorthand for:

fetch('https://pokeapi.co/api/v2/pokemon/', {
  method: 'GET'
})

If you needed a POST request as opposed to a GET request, you could add the extra information as keys on the object, like in the following example:

fetch('https://my-backend.com', {
  method: 'POST',
  data: JSON.stringify(data),
  contentType: 'application/json'
})
The Problem with Promises and fetch()

“If fetch and promises are so great, why hasn’t every developer started using them exclusively,” you might ask. This is because, like most of the other newer functionalities you’ve learned about, older browsers don’t support them—in this case, Safari 10, Internet Explorer 11, and their precursors. According to Can I Use, this means that around 5% of all internet users won’t have access to promises.

But with the growing adoption of newer browsers, it’s only a matter of time before the lack of support stops being an issue. In the meantime, a sneaky little workaround has taken off in recent years that allows you to use promises and fetch in your code while still supporting older browsers: the polyfill.

Polyfills

A polyfill is a piece of code that mimics newer JavaScript features for older browsers.

Once added to your code, a polyfill will check whether certain features (e.g., promises or fetch) are supported by the user’s browser, and if it’s not, it will manually add functions with the same name and implement them with available features.

If your browser doesn’t support promises, for instance, you could simply use a promise polyfill to add this functionality for you. This allows you to use as many new features as you want without worrying about whether or not the older browsers will support the functionality—your polyfill is doing all the hard work for you.

As with everything, however, this comes with a word of warning: polyfills can quickly make even a small app quite large. The code for a polyfill is automatically downloaded by every user that accesses the app, whether they need it or not. That’s a lot of excess bulk, which leads to larger app sizes and potential lag. For this reason, only use them if you have to, and once you no longer need to support those older browsers, simply get rid of them.

There are a number of different polyfills available for promises. The one we recommend is small and simple and is called promise-polyfill. Getting it set up in your app is simple enough:

  1. Download the latest version of promise-polyfill and save it in your app as “js/promise-polyfill.js”.
  2. Include the script tag for your polyfill in your HTML right before your main “scripts.js” file. The order is important! You must load the polyfill before the scripts file, otherwise the polyfill won’t be loaded at the time you’re executing your promises inside the “scripts.js” file. Check the image below for the correct order:

And that’s it! You can now use new Promise(...) with the confidence that it will work on any browser a user may try to access your app with.

There’s a similarly simple polyfill out there for fetch, as well: the fetch polyfill. Do note, though, that you need to add the promise polyfill first for it to work. You can download the fetch polyfill here and include it the same way you included the promise polyfill up above. Again, make sure to get the order right. Load promises first, then fetch, then your own “scripts.js” file.

Promise and fetch polyfills aren’t the only ones out there. A lot of newly released JavaScript functions have their own polyfills, as many of them aren’t yet (or will never be) available on older browsers. But as mentioned before: think twice before you start snatching them up!

If it’s a polyfill for a function you’ll only use once or twice, it’s probably not worth it. If you’ll be using the function many times throughout your code, however, such as promise or fetch, it’s likely worth the added bulk to your app. The final decision lies with you as the developer! Just remember that any additional code you add to your app brings with it a penalty in performance. You’ll need to evaluate the file size of your scripts and weigh them against the benefits they’ll bring for your project.

Figuring Out Browser Support
We’ve talked pretty extensively about features not supported by some browsers. But it’s not just older browsers that cause support problems; current browsers can vary in how they adopt certain features. So, how are you supposed to know what you can and can’t use?

  • The documentation pages of MDN Web Docs always indicate the current browser support for the features in question. Check out, for example, the current support for fetch. (Keep in mind that even the newest browsers often don’t support experimental or extremely new features.) Another place to look is a site called Can I Use, which lets you check if you can use a given feature. Check out the Can I Use pages for fetch and for promises.

Using Fetch in Your Data Repository

Now that you’ve got all this new knowledge to work with, it’s time to see how you can apply it by adding fetch to your Achievement project! Keep in mind that some of these instructions may vary depending on the kind of data you want to use for your app. In this case, you’ll be working with the pokeapi.

First, visit the pokeapi documentation to familiarize yourself with what it is and how it works. Don’t worry if some of it still looks confusing! We’ll be going through it step-by-step!

The pokeapi organizes information on a variety of different Pokémon game elements such as berries, moves, and locations. For your app, the focus is on the Pokémon themselves, so let’s bring up a list of all your data entries.

Go ahead and open up this list of codified Pokémon, which allows you to fetch one or many Pokémon from the database. You should see a list of codified Pokémon on your screen; however, notice that the list only shows 20 (even though there should be 150)? This is because the API is paginated by default. You can change this by adding “?limit=150” to the URL to show that you want to have the full 150 rather than a list of 20. The full link would then appear as: “https://pokeapi.co/api/v2/pokemon/?limit=150”.

Note that this might be slightly different for other APIs. Some, for example, might not use any pagination at all, while for others, it might be called something different, such as “?pageSize=xxx” or “?count=xxx”.

TIP!
Pagination is a very common pattern in apps. You’ll likely have seen it on any blog or online shop you’ve visited. It means the entire data set is split into groups with an even number of items. In traditional web apps or blogs, each group was displayed on its own page, and at the bottom of the page you could switch between the different pages to view the next data set. While modern applications no longer have pages, this pattern is still known as pagination. Apps such as Twitter and Facebook, for example, mask their pagination by automatically loading the next set of data as you scroll down.

Let’s continue. You now have a list of 150 Pokémon names with one link per Pokémon. You’ll want more than that for your app, however. For instance, take a look at all the information listed under the first entry, “Bulbasaur,” that loads with the link: “https://pokeapi.co/api/v2/pokemon/1/”.

TIP!
That mash-up of code is pretty hard to read, isn’t it? Try installing an extension like JSON Viewer for Chrome to help you view JSON files in your browser. This will automatically format JSON files so that they’re much easier to read and explore.

Of course, this is probably a bit too much for your app. In fact, you only want a few pieces of information per Pokémon:

  • Name
  • Image URL

You can also find types and height in the list, the two properties you added to your pokemonListbefore.

But in order to display a list of all the Pokémon along with their name and image, you’d need to:

  1. Load the list of Pokémon
  2. Load the details for each Pokémon

For 150 Pokémon, this would mean you’d need to make 151 requests to the API. This is a huge number of requests and will slow down your app. In cases like these, you should rethink how your app is going to work—that way, you can avoid having to make all those requests. In this case, you don’t have any control over the API you’re using, so you can’t make any changes there.

You can, however, redesign your app to only show the name of the Pokémon on the list itself, and then bring up details (such as the image) when you click on a Pokémon. In general, the less information requested, the faster the response times. In this case, you split the response times over two requests (one for the list items, another—based on click—for the image) to make the app seem a bit faster.

In a different situation where you do have some control over an API, you might be able to redesign the “https://pokeapi.co/api/v2/pokemon/?limit=150” endpoint to also include the image for each Pokémon next to its name.

For now, though, let’s go ahead and add two functions to your repository: loadList() and loadDetails(). First, let’s load the list of Pokémon. Assume that you’ve already added the promise and fetch polyfills, as described above. Then, add the following code inside the repository:

let pokemonRepository = (function () {
  let pokemonList = [];
  let apiUrl = 'https://pokeapi.co/api/v2/pokemon/?limit=150';

  // Other functions remain here

  function loadList() {
    return fetch(apiUrl).then(function (response) {
      return response.json();
    }).then(function (json) {
      json.results.forEach(function (item) {
        let pokemon = {
          name: item.name,
          detailsUrl: item.url
        };
        add(pokemon);
      });
    }).catch(function (e) {
      console.error(e);
    })
  }

  return {
    add: add,
    getAll: getAll,
    loadList: loadList
  };
})();

pokemonRepository.loadList().then(function() {
  // Now the data is loaded!
  pokemonRepository.getAll().forEach(function(pokemon){
    pokemonRepository.addListItem(pokemon);
  });
});

The LoadList() method will fetch data from the API, then add each Pokémon in the fetched data to pokemonList with the add function you implemented earlier. In our example, we want each item to have a name and a detailsUrl property. You can then use the detailsUrl property to load the detailed data for a given Pokémon. For this, you add a loadDetails() function, which takes a Pokémon item as an argument:

let pokemonRepository = (function () {
  // Other functions here…

  function loadDetails(item) {
    let url = item.detailsUrl;
    return fetch(url).then(function (response) {
      return response.json();
    }).then(function (details) {
      // Now we add the details to the item
      item.imageUrl = details.sprites.front_default;
      item.height = details.height;
      item.types = details.types;
    }).catch(function (e) {
      console.error(e);
    });
  }

  return {
    add: add,
    getAll: getAll,
    loadList: loadList,
    loadDetails: loadDetails
  };
})();

In your app, you can use your existing showDetails() function to execute loadDetails(). Remember, the showDetails() function is executed when a user clicks on a Pokémon. That’s the moment when you want to get the Pokémon’s details from the server. This is what your function could look like:

function showDetails(pokemon) {
  loadDetails(pokemon).then(function () {
    console.log(pokemon);
  });
}

With this, you should get a Pokémon’s details logged to the console upon clicking its button.
